## 一、Demo 简介

**是什么：** 知源 KnowSource 是一款类 VSCode/Obsidian 的科研知识工作台桌面应用（Electron + Vue 3），帮助科研人员高效管理多格式文献、构建跨文档知识图谱、记录研究灵感。

**面向谁：** 需要阅读大量论文、追踪研究前沿的科研人员、研究生和博士生。

**主要功能：**

1. **多格式文献管理** — 支持 PDF / Word / PPTX / Markdown / OCR 扫描件，自动解析章节结构，原始 PDF 视图与文本视图双模式切换。一次导入多篇论文，为跨文档知识网络构建打下基础。

2. **跨文档知识图谱** — 基于 TF-IDF + 语义向量的术语抽取（无监督路径），结合 16 个 LLM 任务的增强路径，自动构建跨文档知识网络。3 篇论文的概念自动连接，发现论文间的隐性关联。力导向可视化，节点可拖拽、可编辑。

3. **LLM 透明化（提示词实验室）** — 所有 16 个 LLM 任务的提示词模板运行前即可查看、编辑、保存。运行测试后展示渲染后的提示词和模型响应，每次调用都写入日志可回溯。

## 二、Demo 创作思路

**灵感来源：**

做科研时最痛苦的环节不是读论文，而是读完之后。一篇篇 PDF 堆在文件夹里，读完就忘，论文之间有什么联系全靠脑子记。Zotero 管文件但不建联系，Obsidian 建笔记但不读 PDF，Connected Papers 看引用关系但不能编辑。我想要一个工具：**既能读论文，又能自动构建知识网络，还能随手记灵感。**

**想解决的问题：**

- **论文孤岛问题**：读过的论文之间缺乏可视化关联，概念交叉点难以发现。3 篇关于深度学习、卷积神经网络、注意力机制的中文综述论文，手动找交叉概念需要逐篇对照，而知源自动发现了 60+ 条跨文档连接
- **知识沉淀困难**：阅读时产生的灵感没有与具体论文段落绑定，事后无法追溯
- **LLM 黑盒问题**：现有 AI 工具调用 LLM 的过程不透明，用户不知道提示词是什么、模型返回了什么，无法调试和优化

**为什么做这个方向：**

科研工具市场存在明显的空白：传统文献管理软件（Zotero/EndNote）停留在文件管理层面，AI 论文工具（Elicit/Scite）是黑盒 SaaS，知识管理软件（Obsidian/Notion）不原生支持 PDF 阅读。知源试图填这个空白——**把论文阅读、知识图谱、LLM 透明化三者打通，让科研人员的知识真正形成网络而非文件堆。**

## 四、TRAE 实践过程

### 开发环境

- **开发工具**：TRAE IDE
- **技术栈**：Electron 28 + Vue 3.4 + Vite 5 + D3.js + PDF.js
- **LLM 集成**：Ollama / OpenAI 兼容（DeepSeek / 通义千问 / 智谱 GLM / SiliconFlow）

### 开发关键步骤

#### 步骤 1：项目重构与方向聚焦（2026.06.25）

在 TRAE IDE 中使用 AI 智能体重新梳理项目目标。通过对话让 AI 分析了整个代码库，生成了项目状态报告，识别出过时的架构文档和偏离的核心目标。最终将方向从"通用知识管理"聚焦到"科研人员多文献阅读"场景，删除了 9 份过时文档，重写了 PROJECT_GOALS.md。

> 截图说明：此处放置 TRAE IDE 中项目重构对话的截图，展示 AI 分析代码库并生成状态报告的过程。

#### 步骤 2：提示词系统重构（2026.06.28）

原有架构中，16 个 LLM 任务的提示词模板在运行时才能捕获，用户打开提示词实验室看不到任何内容。在 TRAE 中通过 AI 规划了完整的重构方案：将提示词从"运行时捕获"改为"静态模板 + 直接编辑"架构，集中定义在 `core/prompts/templates.js`，使用 `{{变量名}}` 占位符语法，应用启动后立即可用。

> 截图说明：此处放置 TRAE IDE 中提示词系统重构的代码对比截图。

#### 步骤 3：系统性 Bug 修复（2026.07.03）

通过 TRAE 的代码审查功能，系统性修复了 90+ 个 Bug，包括：
- LLM 解析器 JSON 对象处理缺陷导致 `.map()` 崩溃

- 向量库非原子写入导致数据损坏

#### 步骤 5：PDF 渲染与图谱质量优化（2026.07.05）

修复了 5 个关键问题：
1. **PDF 加载失败**：前端 HEAD 预检请求返回 404 → 后端添加 HEAD 支持
2. **文本编码乱码**：PDF.js 提取的 UTF-8 字符被当作 Latin-1 → 添加 `fixLatin1Mojibake` 函数

4. **标题节点缺失**：规则提取仅 1 个标题 → 回退使用 PDF bookmarks（22 个）
5. **节点质量低**：停用词不足 → 扩充 60+ 英文停用词

## 五、经验总结与开发心得

### TRAE 带来的价值

1. **代码审查能力**：TRAE 的 AI 能系统性扫描整个代码库，一次发现 90+ 个 Bug，包括隐蔽的竞态条件和边界情况，这是人工审查很难覆盖的。

2. **架构重构规划**：提示词系统重构涉及 5+ 个核心文件的协调修改，TRAE 的 AI 智能体能够规划完整的重构方案，保证了修改的一致性。

3. **快速迭代**：从发现问题到修复验证，一个循环通常在几分钟内完成。TRAE 的对话式开发让"描述问题 → AI 修复 → 测试验证"成为自然的工作流。

### 踩坑经历

1. **PDF.js 编码问题**：PDF.js 提取文本时，某些多字节 UTF-8 字符被逐字节当作 Latin-1 返回，导致中文"第1页"变成"ç¬¬1é¡µ"。解决方案是检测 Mojibake 模式后将字节重新组合为 UTF-8。

2. **图谱节点字段丢失**：pipeline 中重建节点时，`label` 被转换为 `content`，但顶层 `label` 属性被丢弃，导致前端显示空白。这种数据流中的字段丢失很难排查，需要从 pipeline 到 storage 到 API 全链路追踪。

3. **跨文档连接的挑战**：3 篇中文论文的术语抽取需要统一的停用词过滤标准——中文论文中"等人提出""计算机应"等碎片词不是概念但容易被 TextRank 提取，英文期刊元数据如"computer""applications"也需精确过滤。经过多轮扩充停用词列表和正则过滤，最终在演示页面中达到节点质量与跨文档连接的平衡。
